Node.js is Node.js. Not Cancer, and not Magic.

 -By Nick Tkachov

So Ted Dziuba wrote an article on why node.js is terrible arguing that "non-blocking io" still blocks and running a cpu intensive loop is going to stall every other operation.

Then Joshua Kehn wrote an article about how node.js is faster than PHP and python at running Dziuba's fibonacchi sequence.

They both missed the point.

Node.js isn't made for running cpu intensive applications inside a single thread. Node.js is good for quick operations that would normally require blocking calls or long operations that wait/spin/idle for a while.

I've written servers in Node.js and Java. Both of my servers run off a single EC2 node (I'm a dirt poor college student). My Java server powers Zombie Outbreak which requires loads of blocking IO calls to all the hash tables that store user data. Zombie outbreak needs to be concurrent though. The back-end logic of the game requires that users be able to upload data while the server does checking, management, and updating. However, whenever the server needs to update the hash tables with new data (such as a zombie infection), it has to block. This operation also makes calls to Amazons simpleDB which, if written in Node.js, could end up blocking connections and cause timeouts with the way the server currently handles these events. Structured differently though, Node.js might be a better choice for this server than Java is.

This blog, however, is written in Node.js. Since each blog post is tiny and each request is fairly simple, Node.js is wonderful for something like this. Node is fast enough to serve each request with enough gusto to not bring the entire blog down. For running simple operations that don't tie up the server, Node makes my job a lot easier by avoiding thread concurrency and race conditions.In all fairness, there is no architectural need for Node.js with this blog, I just wanted to learn javascript and toy with Node.

The other benefit of node.js, although I don't particularly use this feature, is that if you have a lot of downtime on certain operations (Think Thread.sleep()), Node.js may save you. No matter how hard you try, Spawning 400 threads waiting on some distant database, API, or long poll request to return a query is doom. Node solves this exact problem with their non-blocking IO formula. Each Thread you have idling might not take up any CPU time but does take up memory and the overhead for spawning threads may start to slow you down. Node.js doesn't spawn threads or processes so if you have many idling threads waiting for network response Node.js might save you from having to manage a thread pool.

Given 100 programmers and 100 programs you will end up with 10,000 solutions. I'm not arguing that Node.js should be your breakfast, lunch, and dinner. Node.js is a tool. And like my shop teacher said: "Use the right tool for the job. Anything else is a waste of time". Use your head when building your websites. And if you don't like Node.js don't use it.

tl;dr: Node is good for long, idleing requests and short quick requests. If you don't like it, don't use it.

Daily Nerd Tech Stack

thedailynerd.com was built entirely from the ground up using

  • Node.js
  • HTML
  • CSS
  • Javascript
On top of Amazon's EC2 platform. Node handles everything from generating any search results (which includes the front page, as a request to index.html results in a search query for posts 1-10), to serving css and javascript files in the background.On boot, the server will access our Amazon S3 bucket and automatically pull its template files and parse them. Any requests made for any files from the server automatically hit the S3 bucket if the server doesn’t find them already loaded in memory. Though it will load any files that are hit into memory as soon as it retrieves them from S3 so further requests for a file will always hit memory instead of S3.

Inside node our site is equipped to handle a few different requests.

  • “?num=#:#” request (where ‘#’ is a number) will return the posts in order from the first number to the second number(newest to oldest). If the first number is larger than the second number it will return all the posts in reverse order (oldest to newest).
  • “?post=posts/.html” will return a post that is located in that path. All posts come from memory as dictated by a “posts.xml” file on the server. So it will only pull known files from the server.
  • “?where=tag” will return any posts marked with what ever tag is requested. Simple as that. If I mark a post “funny” “?where=funny” will return that post (and any other posts i marked funny).
  • ”?tags=post/.html” will return the tags for a given post. It’s not particularly useful at the moment and I haven’t implemented them into site yet but hopefully they will eventually show (maybe in a popup) what a given post is tagged
That’s about it for this site. More features will come with time as I develop the engine that powers it. I will update this post with a github link to the source of the entire engine so that anyone can use it.
